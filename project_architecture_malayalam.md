ഈ പ്രൊജക്റ്റിലെ പ്രധാന ഫയലുകളും ഫോൾഡറുകളും, അവയുടെ ഘടനയും (Architecture) താഴെ വിശദീകരിക്കുന്നു. ഉപയോഗശൂന്യമായ ഫയലുകൾ നീക്കം ചെയ്യുന്നതിന് മുൻപ് ഈ ഘടന മനസ്സിലാക്കുന്നത് വളരെ പ്രധാനമാണ്.

## 🏛 2. ആർക്കിടെക്ചർ മനസ്സിലാക്കാം (Understanding the Architecture)

ഈ പ്രൊജക്റ്റ് **Next.js App Router**, **MongoDB**, **TypeScript** എന്നിവ ഉപയോഗിച്ചാണ് നിർമ്മിച്ചിരിക്കുന്നത്. 

### 1. Frontend (ഫ്രണ്ട്-എൻഡ് / യൂസർ ഇന്റർഫേസ്)
* `src/app/` ഫയൽ സിസ്റ്റം ആണ് ഇവിടെ റൗട്ടിംഗ് (Routing) നിയന്ത്രിക്കുന്നത്.
* ഇതിൽ പ്രധാനമായി **3 ഭാഗങ്ങൾ** ഉണ്ട്:
  1. **Admin Section (`/admin`)**: അഡ്മിന് മാത്രം കയറാൻ പറ്റുന്ന പേജുകൾ (മത്സരങ്ങൾ ആഡ് ചെയ്യാനും മറ്റും).
  2. **Site Section (`/site`)**: ലോഗിൻ ചെയ്ത യൂസേർസിന് മത്സരങ്ങൾ കാണാനും പ്രവചിക്കാനുമുള്ള പേജുകൾ.
  3. **Public Section**: `app/login/page.tsx` വഴിയാണ് യൂസേർസ് രജിസ്റ്റർ ചെയ്യുന്നതും ലോഗിൻ ചെയ്യുന്നതും.

### 2. Backend API (API റൂട്ടുകൾ)
* `src/app/api/` എന്ന ഫോൾഡറിനുള്ളിൽ ആണ് ഇതിന്റെ ബാക്കെൻഡ് കോഡ് നിലനിൽക്കുന്നത്.
* ഉദാഹരണത്തിന്, യൂസർ പ്രവചനം നടത്തുമ്പോൾ, ഫ്രണ്ട്-എൻഡ് `/api/predictions` എന്ന റൂട്ടിലേക്ക് കോൾ ചെയ്യും.

### 3. Business Logic (ബിസിനെസ്സ് ലോജിക്)
* നേരിട്ട് API റൂട്ടുകളിൽ കോഡ് എഴുതുന്നതിന് പകരം, സങ്കീർണ്ണമായ ഡാറ്റാബേസ് ലോജിക്കുകൾ എടുത്ത് **`src/services/`** ഫോൾഡറിലേക്ക് മാറ്റുന്നു (Service Layer Architecture). 
* ഇതിലൂടെ കോഡ് വായിക്കാൻ എളുപ്പമാവുകയും മെയിൻ്റനൻസ് എളുപ്പമാവുകയും ചെയ്യുന്നു.

### 4. Database Schema (ഡാറ്റാബേസ് രൂപരേഖ)
* **`src/models/`** ഉള്ളിലാണ് നമ്മൾ ഡാറ്റാബേസ് പ്ലാൻ വരച്ചിരിക്കുന്നത്.
* `User` (യൂസർ വിവരങ്ങൾ), `Match` (മത്സരങ്ങൾ), `Prediction` (പ്രവചനങ്ങൾ), `OtpStore` (OTP സൂക്ഷിക്കാൻ) എന്നിവയാണ് പ്രധാന മോഡലുകൾ.

## 🗑 3. എന്തിനാണ് നമ്മൾ ചില ഫയലുകൾ കളയുന്നത്? (Why remove some files?)

നമ്മുടെ കോഡിൽ നിലവിൽ ചില കാര്യങ്ങൾ രണ്ടുതവണ (duplicates) എഴുതിയിട്ടുണ്ട് അല്ലെങ്കിൽ പഴയ കോഡ് ഡിലീറ്റ് ചെയ്യാതെ ഇട്ടിട്ടുണ്ട്.

**എന്തിന് മാറ്റണം?**
1. ഡ്യൂപ്ലിക്കേറ്റ് കോഡുകൾ പ്രൊജക്റ്റിൽ കൺഫ്യൂഷൻ ഉണ്ടാക്കും.
2. ഹാക്കിങ്ങിനോ സെക്യൂരിറ്റി പ്രശ്നങ്ങൾക്കോ ഇത് കാരണമായേക്കാം.
3. വെബ്സൈറ്റിന്റെ വേഗതയും പ്രകടനവും (Performance) കൂട്ടാൻ.

**കളയാൻ പോകുന്ന പ്രധാന ഫയലുകൾ:**
* `Participant.ts` (നമ്മൾ ഇതിനകം `User.ts` ഉപയോഗിക്കുന്നുണ്ട്, അതിനാൽ ഇത് വേണ്ടാത്ത ഫയലാണ്).
* `send-otp/route.ts` & `verify-otp/route.ts` (നമ്മുടെ ശരിയായ ലോഗിൻ സിസ്റ്റം `/api/auth/login`, `/api/auth/register`, `/api/auth/verify` എന്നിവ വഴി ഭംഗിയായി പ്രവർത്തിക്കുന്നുണ്ട്).
* അതുപോലെ ഉപയോഗിക്കാതെ കിടക്കുന്ന മറ്റ് സർവീസുകളും ലൈബ്രറികളും (`sendoxi-otp.ts`, `session.ts` എന്നിവ).
